# ERRORS — Аналіз та Вирішення (2026-03-04)

## ✅ Проблема 1: Текст не вставлявся автоматично (ВИРІШЕНО)

### Симптоми
- Deepgram розпізнавав мову, але текст не з'являвся в активному вікні.
- Ручне Ctrl+V працювало, але автоматичне — ні.

### Рішення
- **Threading Fix**: Перенесено виклик вставки тексту з фонового потоку (Deepgram) у головний потік Qt за допомогою `QTimer.singleShot(0, ...)`. Це гарантує, що Windows API коректно обробляє емуляцію клавіш.
- **Win32 API Fix**: Замінено `pyautogui` на прямі виклики Win32 API (`keybd_event`) через `ctypes`. Це обходить обмеження системи безпеки Windows (UIPI), які блокували вставку.
- **Focus Fix**: Видалено виклики `pyautogui.keyUp('win')`, які раніше викликали відкриття меню "Пуск" і забирали фокус з вікна.

## ✅ Проблема 2: Deepgram не отримував аудіо (ВИРІШЕНО)

### Симптоми
- Помилки `1011: Deepgram did not receive audio data`.
- Текст не повертався взагалі, що робило вставку неможливою.

### Рішення
- **Streaming Enable**: У `tray_app.py` додано примусове ввімкнення режиму стрімінгу (`controller.enable_streaming(True)`). Раніше він був вимкнений за замовчуванням, через що аудіо-чанки не відправлялися.

---

## ✅ Проблема 3: Програма не запускалась автоматично (ВИРІШЕНО)

### Симптоми
- Після перезавантаження Windows програма не з'являлася в треї.

### Рішення
- **Registry Fix**: Змінено запис у реєстрі з `.bat` файлу на прямий виклик `pythonw.exe` з абсолютними шляхами.
- **Path Resolution**: Виправлено визначення кореневої папки проекту в `os_utils.py` через `__file__` замість `sys.argv[0]`.

---

## ✅ Проблема 4: Диктовка переривалася на коротких паузах (ВИРІШЕНО)

### Симптоми
- При паузі більше 0.3с Deepgram фіксував речення, що заважало довгій диктовці.

### Рішення
- **Endpointing Fix**: Збільшено параметр `endpointing` з `300` до `1000` (1 секунда). Це дозволяє робити природні перерви під час мовлення без розриву сесії.

---

## ✅ Проблема 5: Паралельні сесії та помилка 1011 (ВИРІШЕНО)

### Симптоми
- При швидкому натисканні клавіш створювалося кілька WebSocket-з'єднань одночасно.
- Зайві з'єднання таймаутили з помилкою `1011 (internal error)`, що заважало основній сесії.

### Рішення
- **Session Guard**: Додано перевірку в `Transcriber.start_streaming`. Якщо сесія вже активна або потік запускається, новий запит ігнорується.

---

## ✅ Проблема 6: Краш потоку через некоректні параметри (ВИРІШЕНО)

### Симптоми
- Після додавання `no_delay` та `utterance_end_ms` транскрипція перестала працювати (thread crash).

### Рішення
- **Compatibility Fix**: Видалено параметри, що не підтримуються поточною версією SDK/API. Повернуто стабільну конфігурацію обробників подій (`MESSAGE`, `ERROR`).
